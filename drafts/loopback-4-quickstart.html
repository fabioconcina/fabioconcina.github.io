LoopBack 4 Quickstart

April 13, 2019

LoopBack is a Node.js API Framework developed and maintained by StrongLoop, a company acquired by IBM in 2015.
The framework empowers the user to automatically generate APIs based on a model, connect them to backend data sources and expose the endpoints over HTTP to be called from a client.
The latest release (LoopBack 4) leverages TypeScript, has full support for Promise based APIs and async/await programming and is OpenAPI Specification compliant.
The framework comes with a CLI interface that makes very easy to scaffold a project for standard CRUD operations, leaving room for extensibility if needed.

The framework has some key components:

  • Application: central class for setting up all module's components, controllers, servers and bindings.
  • DataSource: named configuration for a Connector instance that represents data in an external system. It comes with an associated .json configuration file to set connection parameters.
  • Model: a description of the shape of the data, with a list of properties like name, type and constraints.
  • Repository: a specialized Service interface that provides strong-typed data access operations of a domain model against a data source.
  • Controller: a class that implements the application business logic and bridges the HTTP endpoints with the models.

To start using LoopBack 4, you have to install Node.js (v8.9 or higher).
To use the CLI provided with the framework, install it globally with npm:

npm i -g @loopback/cli

To generate a new project, run from command line:

lb4 app

The interface will guide you in the setup of the project:

Once finished, you can cd into the project folder and you can already run the application using npm start.
By default, the app will be served on http://localhost:3000 and it will present a home page with two links:

The route /openapi.json serves a static JSON file with the OpenAPI description of the REST API.
The route /explorer serves an API explorer that lists the available routes and allowed methods, their description and an interactive UI to try them out.
The project comes with an example route already defined, which is the /ping route served by PingController. The response body is a JSON containing the current date and some additional info.

We can now start to add new routes to the application.
Suppose you have a simple table on a relational database, for example PostgreSQL, with the following definition:

create table drone (
    id bigserial primary key,
    name text not null,
    brand text,
    weight real,
    max_speed real,
    has_camera boolean
);

First, we need to define the connection to the database. To do that, we create a DataSource:

We set the id field as not required, as it will be generated via PostgreSQL sequence and we don't want to pass it as a parameter.
We can find the created .ts file under src/models/:

import {Entity, model, property} from '@loopback/repository';

@model()
export class Drone extends Entity {
  @property({
    type: 'number',
    id: true,
  })
  id: number;

  @property({
    type: 'string',
    required: true,
  })
  name: string;

  @property({
    type: 'string',
  })
  brand?: string;

  @property({
    type: 'number',
  })
  weight?: number;

  @property({
    type: 'number',
  })
  max_speed?: number;

  @property({
    type: 'boolean',
  })
  has_camera?: boolean;


  constructor(data?: Partial<Drone>) {
    super(data);
  }
}

We can now start the app using npm start and use the defined APIs through the Explorer interface:

Suppose we want to get the list of all the records on the database; we can use the UI to select the route GET /drones. Expanding the tab we see the route takes no parameters and gives as response a JSON containing the list of records, for example:

[
  {
    "id": 1,
    "name": "Spark",
    "brand": "DJI",
    "weight": 300,
    "max_speed": 50,
    "has_camera": true
  },
  {
    "id": 2,
    "name": "Mavic",
    "brand": "DJI",
    "weight": 734,
    "max_speed": 65,
    "has_camera": true
  }
]

Next, we can try to add a new drone to the list, using the same route but with a POST method. We set a JSON with the new record in the request body:

{
  "name": "Phantom 4 Pro v2",
  "brand": "DJI",
  "weight": 1375,
  "max_speed": 72,
  "has_camera": true
}

We can now call /drones/count to verify that a new record has been added to the list:

{
  "count": 3
}

This only scratches the surface of what can be created using the framework, but we already have a fully operational REST API which we can for example wire up with a frontend application.
